# Spec Generation

Reverse-engineer `design.md` + 5 category files from a live site when no specs exist. Entry point is `verification.md` step 2 (URL given, no design.md, user confirms generation). Output paths default to `<project_root>/visual/design.md` and `<project_root>/.ui-verification/specs/<category>.md`.

## Phase 0: Detect source code (do this BEFORE opening the browser)

Generation gets dramatically more accurate when the user has source for the running app. Tokens become named (`{colors.brand-primary}` instead of `~rgb(R,G,B)`), components become typed (`<PrimaryButton>` instead of `button:nth-of-type(2)`), and selectors are grounded in actual class/attribute usage rather than DOM observation.

**Detection rule:** look for `package.json` + `src/` in the default locations:
- At `output_dir` (project root layout)
- One level deeper, e.g. `output_dir/<package-name>/package.json` (workspace layout — the verifier opens at the workspace root and one or more app packages live as subdirs)

If multiple candidate packages exist, prefer the one whose name or `dev`/`start` script matches the URL being verified (e.g. `npm run dev` on port 5173 → match against the localhost URL). If still ambiguous, or if no `package.json` is found in either default location, **ask the user once: "Where is the source root for the app at `<URL>`? (relative path from `<output_dir>`, or `none` for DOM-only mode)"** Do NOT search the filesystem broadly — never `find /`, never search outside `output_dir`. One question, then take the user's answer or fall back to DOM-only.

**When source is detected**, before Phase 1:
1. Read `package.json` to identify the framework (React, Vue, etc.) and design-system dependencies (`bootstrap`, `tailwind`, `styled-components`, `@mui/material`, etc.).
2. Read theme / token files: `theme.ts`, `theme.tsx`, `tokens.json`, `tailwind.config.js`, `Config/*.ts`, `src/styles/`, `src/theme/`. These give you NAMED tokens — preserve the names in `design.md`'s YAML frontmatter rather than assigning made-up names to RGB values you observed.
3. Skim `src/components/` (or equivalent) for the top-level component names — these become candidate `## Components` entries in `design.md`.
4. Note global CSS (`src/index.css`, `src/styles.css`, etc.) — global resets, body styles, font imports.

**When source is NOT detected** (external site, no source access), proceed to Phase 1 — generation runs in pure DOM-only mode. The output `design.md` will use observed RGB values for token names (e.g. `primary: "rgb(255,56,92)"`), which the user can rename to brand names during review.

**Source-aware generation produces better `design.md`:**

| Without source | With source |
|---|---|
| `colors.value-rgb-R-G-B: "#hexcode"` (made-up name) | `colors.brand-primary: "#hexcode"` (named from theme) |
| Selector: `button.css-1abc23` (autogenerated class — fragile) | Selector: `button.btn-primary` (stable class from CSS module) |
| Component: "primary action button" | Component: `<PrimaryButton>` (matches source filename) |
| Tokens that are *applied* to DOM only | All defined tokens, including unused-yet-named ones |

## Generation Workflow

The workflow uses `screenshot` + `get_page_content` for **agent-side reasoning** (you, the agent, perceive the page using your own multimodal model — Opus, Sonnet, etc.), `act` to drive the browser, `act_get` for structured data extraction, and `evaluate_js` for precise computed style values. Build all 5 category files incrementally as you explore — don't wait until the end.

In source-aware mode, also cross-reference what you observe in the DOM against the source's token names and component definitions — when an observed RGB matches a token in the theme file, USE the token name in `design.md` rather than re-deriving a name from the RGB value.

The browser session MUST stay open across all phases. If it closes mid-flow, restart with the same URL and `browser_mode="local"`.

**Tool roles — read this carefully before Phase 1:**

| Tool | What it does | Who reasons |
|---|---|---|
| `screenshot(session_id)` | Captures the current viewport as an image | **The agent.** You look at the image yourself with your own multimodal model. |
| `get_page_content(session_id, format)` | Returns page HTML or text | **The agent.** You read the HTML and reason about structure. Fallback when `act_get` doesn't fit. |
| `act(session_id, prompt)` | Instructs nova-act to *do* something — scroll, click, fill a form, navigate. Multi-step + spatial hints OK. | Nova-act executes. You wrote the instruction. |
| `act_get(session_id, prompt, schema?)` | Structured data extraction with optional JSON schema (e.g. extract `[{title, price}, ...]` from a product list) | Nova-act extracts. You consume the structured output. |
| `evaluate_js(session_id, script)` | Runs JS, returns computed values | The script. Use for precise style values, never for "describe this page." |

**Anti-pattern (do NOT do this):**

```
act_get(session_id, "Describe the main UI areas visible on this page")
act_get(session_id, "What buttons and interactive elements are visible?")
act_get(session_id, "What navigation structure does the header have?")
```

These are perception/reasoning prompts, not structural extraction. You are the agent — you have a multimodal model. Take a screenshot, get the HTML, and reason about layout, components, and structure yourself. Asking nova-act to "describe what's visible" outsources reasoning to a tool that's there to *act*, not to perceive on your behalf.

### Phase 1: Visual Exploration (you reason from screenshot + DOM)

Build a mental model of the page using **your own multimodal reasoning** over `screenshot` + `get_page_content`. Use `act` only to drive the browser (scroll, click, navigate). Use `act_get` only when you need structured extraction with a schema.

**Minimum depth — Phase 2 cannot start until this is met:**

You MUST cover at least 3 distinct page regions (header, content/hero, footer at minimum). For each region, you MUST take at least one `screenshot` AND read the corresponding portion of HTML (via `get_page_content` once for the whole page is enough). One screenshot followed by jumping to `evaluate_js` is NOT Phase 1 — you haven't exercised below-fold areas, navigation states, or component variants.

A complete Phase 1 typically includes ≥2 `screenshot` calls (above and below fold), 1 `get_page_content`, and ≥1-2 `act` calls to navigate or scroll. If you find yourself reaching for `evaluate_js` without having scrolled or looked at multiple regions, stop and explore more.

**Goals (you answer these by reasoning over screenshot + HTML):**
- Identify the page sections (header, hero, content areas, footer)
- Identify interactive components (buttons, links, tabs, cards, forms)
- Identify the color palette in use (primary CTAs, backgrounds, text colors)
- Identify the typography scale (headings, body, captions)
- Note navigation patterns (tabs, menus, breadcrumbs)

**Recommended sequence:**

```
screenshot(session_id)                          → look at the viewport
get_page_content(session_id, format="html")    → read the DOM structure
                                                  (you reason about both — header? hero? nav? components?)
act(session_id, "Scroll down to see more of the page")
screenshot(session_id)                          → see below-fold content
                                                  (you continue reasoning about what changed)
```

When you need *structured extraction* (e.g. a list of all visible product card titles + prices), use `act_get` with a schema:

```
act_get(session_id,
        "Extract every product card visible on the page",
        schema='[{"title": "string", "price": "string", "rating": "number"}]')
```

When you need to drive the browser to reveal something (open a tab, expand an accordion, scroll to a known element), use `act`:

```
act(session_id, "Click the second tab in the navigation")
screenshot(session_id)                          → see the new state, reason yourself
```

If `act_get` fails or returns nothing useful, fall back to `get_page_content` (text or HTML) and parse it with your own reasoning.

**Start writing category files as you discover things:**
- See a header with logo + nav + search in the screenshot/HTML? → start `platform-conventions.md` and `accessibility.md`
- See distinct button styles? → start `component-rules.md`
- See color patterns? → note them for targeted `evaluate_js` extraction in Phase 2

### Phase 2: Targeted Extraction (evaluate_js)

Once you know WHAT exists from Phase 1, use `evaluate_js` to extract precise computed values. Target specific areas you identified — don't dump the entire DOM blindly.

**Typography extraction (for elements you identified in Phase 1):**

```
evaluate_js(session_id, "(() => {
  const targets = ['body','h1','h2','h3','p','a','button','nav a'];
  const result = {};
  for (const sel of targets) {
    const el = document.querySelector(sel);
    if (!el) continue;
    const s = getComputedStyle(el);
    result[sel] = {fontFamily: s.fontFamily.split(',')[0].trim(), fontSize: s.fontSize, fontWeight: s.fontWeight, lineHeight: s.lineHeight, letterSpacing: s.letterSpacing, color: s.color, textTransform: s.textTransform};
  }
  return result;
})()")
```

**Color palette extraction:**

```
evaluate_js(session_id, "(() => {
  const targets = ['body','header','footer','button.primary','[class*=card]','nav'];
  const result = {};
  for (const sel of targets) {
    const el = document.querySelector(sel);
    if (!el) continue;
    const s = getComputedStyle(el);
    result[sel] = {backgroundColor: s.backgroundColor, color: s.color, borderColor: s.borderColor};
  }
  return result;
})()")
```

**Component extraction (adapt selectors based on Phase 1 findings):**

```
evaluate_js(session_id, "(() => {
  const components = [/* selectors from Phase 1 observations */];
  const result = {};
  for (const sel of components) {
    const els = [...document.querySelectorAll(sel)].slice(0, 3);
    if (!els.length) continue;
    result[sel] = els.map(el => {
      const s = getComputedStyle(el);
      return {backgroundColor: s.backgroundColor, color: s.color, borderRadius: s.borderRadius, padding: s.padding, height: s.height, display: s.display};
    });
  }
  return result;
})()")
```

**Landmarks and roles:**

```
evaluate_js(session_id, "(() => {
  const roles = {};
  document.querySelectorAll('[role]').forEach(el => {
    const r = el.getAttribute('role');
    roles[r] = (roles[r] || 0) + 1;
  });
  const landmarks = {};
  for (const sel of ['header','main','nav','aside','footer']) {
    const count = document.querySelectorAll(sel).length;
    if (count > 0) landmarks[sel] = count;
  }
  return {roles, landmarks};
})()")
```

### Phase 3: Build category files (incremental — start during Phase 1)

Write to the 5 category files as you go. Don't wait until all extraction is done. After each exploration or extraction step, append rules to the relevant file.

**Build order (natural from exploration):**
1. `platform-conventions.md` — page structure, nav patterns (visible immediately from screenshot)
2. `accessibility.md` — landmarks, roles (visible from first DOM check)
3. `visual-style.md` — colors, typography, spacing (extracted in Phase 2)
4. `component-rules.md` — button variants, cards, inputs (extracted per-component)
5. `project-rules.md` — layout structure, containers, conventions (last, as it requires cross-page understanding)

### Phase 4: Convert to spec rows with correct constraints

Translate raw computed values into spec file claims:

| Computed value pattern | Constraint to use |
|---|---|
| Exact pixel value (`14px`, `32px`) | `=14px` |
| Color with alpha (`rgba(17, 24, 39, 0.6)`) | `~rgb(17,24,39)` — verifier ignores alpha |
| Color without alpha (`rgb(248, 250, 252)`) | `~rgb(248,250,252)` |
| Border-radius percentage (`50%`) | `>=50%` |
| Font-weight numeric (`600`, `700`) | `>=600` |
| Font-family (contains fallbacks) | `contains:Inter` |
| `text-transform: uppercase` | `=uppercase` |
| "Must not be serif" | Use `evaluate_js` — `!contains:serif` false-positives on `sans-serif` |
| Element count / structural fact | Use `evaluate_js` |

**Each spec row must have:**
- `name` — verbatim text that will appear in the spec file (for warm-start diffs)
- `selector` — specific enough that all matched elements share the expected value
- `property` — single CSS property in **kebab-case** (e.g., `background-color`, `font-size`, `border-radius`). The verifier uses `getPropertyValue()` which requires kebab-case — camelCase returns empty string and fails with "no value".
- `constraint` — using the table above

**Property name conversion (JS → CSS):**
The extraction scripts use camelCase (e.g., `backgroundColor`, `fontSize`) because that's how JavaScript accesses computed styles. When writing rules, always convert:
`backgroundColor` → `background-color`, `fontSize` → `font-size`, `fontWeight` → `font-weight`, `fontFamily` → `font-family`, `lineHeight` → `line-height`, `letterSpacing` → `letter-spacing`, `borderRadius` → `border-radius`, `textTransform` → `text-transform`, `maxWidth` → `max-width`, `gridTemplateColumns` → `grid-template-columns`, `alignItems` → `align-items`, `justifyContent` → `justify-content`, `textAlign` → `text-align`, `flexDirection` → `flex-direction`

**Category assignment:**
- Typography (font-family, font-size, font-weight, line-height, letter-spacing, color, text-transform) → `visual-style.md`
- Background colors, border-radius, spacing tokens → `visual-style.md`
- Component-specific rules (button variants, card styles, named components) → `component-rules.md`
- Landmarks, roles, heading hierarchy, aria attributes → `accessibility.md`
- Layout structure (position, display, height, max-width for structural elements) → `project-rules.md`
- Navigation patterns, page sections, content structure → `platform-conventions.md`

**Avoid redundancy across categories:** If a property appears in `visual-style.md` (e.g., nav background color), don't repeat it in `project-rules.md`.

### Phase 5: Sanity-check — verify against the same session

Run the generated rules immediately against the same browser session that produced them:

```
verify_visual_style(session_id, rules_json, output_dir)
verify_components(session_id, rules_json, output_dir)
verify_accessibility(session_id, rules_json, output_dir)
verify_project_rules(session_id, rules_json, output_dir)
verify_platform_conventions(session_id, rules_json, output_dir)
```

**The target is 100% pass rate. Generation = reverse engineering the running app.** There is no upstream `design.md` to anchor against — `design.md` is being CREATED by this run from the running system. The running system IS the intent until the user reviews and edits.

#### Source code's role during generation: NAMES, not VALUES

When source code is accessible, source informs **names** (theme tokens, component names, file-derived structure). The **DOM** is the source of truth for the **value** to record in each constraint.

| Source's role | DOM's role |
|---|---|
| `--surface-canvas: #hexcode` in `tokens.css` → token name `surface-canvas` in `design.md` | `body { background-color: rgb(R, G, B) }` from `getComputedStyle` → constraint `~rgb(R,G,B)` |
| `<PrimaryButton>` component in `src/components/` → `## Components / PrimaryButton` in `design.md` | `.btn-primary` resolves to `padding: 16px 40px` → record `=16px 40px` |
| Theme exports a `radius.lg` of `14px` → token name `rounded.lg` | `.destination-card` renders `border-radius: 14px` → constraint `=14px` |

#### "Source CSS says X, DOM says Y" is NOT a divergence

If you find yourself reasoning "the source says 5rem but the DOM is 4rem — there's a cascade leak," **stop**. There is no divergence. The DOM is the cascade-resolved outcome of **all** source CSS — every file, every specificity, every `!important`, every CSS variable. The "disagreement" you see is between **one CSS file you happened to read** and **the same source compiled by the browser**. Both are the same source code; the browser just read more of it than you did.

The only genuinely different thing the DOM can show vs. all-of-source is JS that mutates styles after layout (rare, intentional). Outside that, "source disagrees with DOM" means you read one rule and missed others.

**During generation, do NOT chase cascade archaeology** ("which file overrides this?", "where does this padding come from?"). Don't `find` for declarations across files. Don't trace specificity. **Just record what the DOM renders.** That value IS the shipped behavior of the source.

#### What to write in `design.md`

- **Token VALUES** come from observed DOM (resolved RGB, resolved px).
- **Token NAMES** come from source theme files when available, otherwise heuristic names from observation.
- **Component definitions** mirror source file/component names when available, otherwise descriptive names from observation.

After generation, the user reviews `design.md` and edits intent where shipped behavior doesn't match what they want. **That edit is when divergence becomes real** — design.md says X, DOM still ships Y, next verification run catches the drift. That's where the verifier earns its keep, not at cold-compile.

#### Optional: Cascade Observations section in the report

If during Phase 1/2 exploration you noticed cross-route or cross-file CSS that surprised you (e.g. `destination-detail.css` declares `.hero-title { font-size: 4rem }` route-globally and overrides `homepage.css`'s home-page-only intent), include a **Cascade Observations** section in the generation report — purely informational. Format:

```markdown
## Cascade Observations (informational)

The following CSS structure may be unintentional. Each item is a place where
the cascade resolves to a value that may not be what one of the source files
declares per-route. Review and decide:

- `.hero-title { font-size }` resolves to 4rem (= `destination-detail.css`,
  `why-go.css`). `homepage.css` declares 5rem but loses cascade because the
  later files set the rule globally rather than scoping to their own routes.
  If the home page should ship at 5rem, scope the overrides to `/destination/*`
  and `/why-go`. If 4rem is correct everywhere, remove the home-page rule.
```

This is FYI for the user. **Does NOT block the integrity ledger.** **Does NOT count as a Phase 5 failure.** The recorded constraint matches DOM (4rem); the cascade observation just tells the user "your CSS architecture has a cross-file rule you might want to review."

#### Phase 5 failures: what they actually mean

Self-check failures during generation are derivation bugs the agent should fix:

- **Selector is too broad** — multiple elements match with different values. Split into scoped selectors.
- **Wrong constraint type** — e.g., exact match `=` when tolerance `~rgb()` is needed; `~rgba()` (engine doesn't accept the `a`) → use `~rgb()`.
- **Property name in camelCase** — verifier's `getPropertyValue()` requires kebab-case (`background-color`, not `backgroundColor`).
- **Stale or flaky selector** — element includes a hidden/disabled variant. Tighten the selector.
- **Extraction bug** — the value was misread. Re-extract with `evaluate_js` for that specific element.

Drop a claim ONLY when:
- The element's value is genuinely dynamic (animation mid-frame, random content).
- Multiple instances exist with legitimately different values and no clean selector can isolate one group.
- The constraint engine cannot express the check without false positives (e.g., `!contains:serif` matching `sans-serif`; `fr` units resolving to pixels — drop the `fr` constraint, keep the structural `display: grid` + `gap` constraints).

Never broaden a constraint to "match anything" (`>=0px`, `contains:`). That's not a spec — it's noise. If a precise constraint can't express what you observed, drop the rule and let the user add it back with a custom check.

Re-run verification after each fix pass. Only write specs to disk when remaining claims all pass honestly.

**Browser-loss STOP rule (critical):**

If the browser session disconnects or `verify_*` calls start failing with connection/MCP errors, STOP immediately. Do NOT:

- Edit spec files based on guesses about what the right selector or constraint should be
- "Tighten" selectors without re-running verify_*
- Drop or rewrite rules to make assumed failures go away
- Make multiple successive edits hoping one will be correct

Spec edits without verification are spec damage. The whole guarantee of generation is that every written claim was confirmed against the live DOM. Lose that, you lose the contract.

What to do instead:
1. Save the partial state — keep the spec files exactly as they were at the moment the browser died.
2. Try ONE reconnect via `start_browse` with the same URL. If it succeeds, resume verification on the unfinished category.
3. If reconnect fails, report to the user: "Generated N rules verified, M rules unverified due to browser disconnect. Specs saved as-is. Rerun spec generation when MCP is healthy to validate the unverified rules."
4. Exit. Do not edit further.

### Write specs and confirm with user

**Output: 6 files total** — `design.md` + 5 category files.

1. **Write `design.md`** — see `references/spec_authoring.md` for the file's structure (prose + YAML token frontmatter + tables). Seed it with the tokens, components, and structure you observed during Phases 1-2.

2. **Write 5 category files to `.ui-verification/specs/`** — these are the verification input. Since you have live DOM access during generation, selectors MUST be validated and precise:
   - You already have the exact DOM structure from `evaluate_js`
   - Selectors should be confirmed to resolve (you can test them in the same session)
   - No guessing or derivation needed — you're looking at the live DOM
   - Each rule should pass verify_* against the current page (sanity check)

   Category files are clean markdown — no frontmatter. Optionally include a one-line provenance comment for human readers:

   ```markdown
   <!-- generated from: https://example.com | 2026-05-18 -->

   # Visual Style

   ## Scope: any
   | Name | Selector | Property | Constraint |
   ...
   ```

3. **Write the integrity ledger.** After all 5 category files are written, compute hashes and write `<output_dir>/.ui-verification/.integrity.json` (atomic write — temp file then `mv`). See `references/spec_sync.md` § Integrity Ledger for the schema and the canonical write recipe. The ledger reflects "Compiler emitted these files from this design.md" — it's about file integrity, NOT about whether the user has reviewed design.md yet.

   **Surface the user-review caveat separately, in the response to the user.** Generation produces both `design.md` (intent layer) and the 5 category files (compiled rules). The ledger is written so future runs can do warm-start, but the user should still review `design.md` before trusting verification results long-term — generation's design.md is observation-derived (what the site looks like), not necessarily what the user *intends*. Recommend: "I generated `design.md` from the live site. The verification results below assume this design.md captures your intent; please review and edit it where it doesn't match what you actually want." This is a workflow nudge to the user, not a reason to skip the ledger.

4. Present a summary to the user:
   ```
   Generated specs from https://your-site.com:

   design.md — human-readable source spec
     Colors: 8 tokens | Typography: 6 scales | Components: 5 variants

   Category files (.ui-verification/specs/):
     visual-style.md — 12 rules (6 colors, 4 typography, 2 spacing)
     component-rules.md — 8 rules (3 button variants, 2 card styles, 3 form elements)
     accessibility.md — 6 rules (5 landmarks, 1 heading hierarchy)
     project-rules.md — 5 rules (header, footer, container layout)
     platform-conventions.md — 4 rules (nav structure, search, footer links)

   Self-verified: 100% pass (N rules total).

   Skipped (could not generate reliable claims):
     - `p` font-family: multiple variants with no clean scoping selector
     (user can add these manually with scoped selectors)

   These represent the CURRENT state of the site. Review design.md and edit
   to match your INTENDED design, then run verification to track drift.
   ```
5. Ask: "Want me to keep these specs as-is, or would you like to review/edit them first?"

**Why design.md matters:** Without it, the 5 category files are opaque rule tables. design.md gives the user a readable document they can edit to express intent — and re-compilation from design.md to category files handles the translation back to rules.

**Why selectors are accurate during generation:** Unlike the verification workflow (which derives selectors from spec descriptions using general web knowledge), generation has the live DOM open. Use it — validate every selector before writing. The selectors in generated category files should be near-perfect because you're reading from the source.

**Important:** Generated specs are a starting point, not a final contract. The user should review and edit before relying on them to catch regressions.

## Honest Limitations

Generation captures what's observable on the visited page. It does NOT capture:

- **Design intent** — only observes current state, can't know "this should be brand-red"
- **Hover / active / focus states** — unless explicitly triggered with `hover()` during Phase 1
- **Responsive breakpoints** — the session uses one viewport size
- **Pages not visited** — only what was explored in this session
- **Animation / transitions** — not observable via `getComputedStyle`
- **Unused design tokens** — only tokens actually applied to DOM elements

Tell the user about these gaps in the summary. The user fills them in during review (rename observed RGB values to brand names, add coverage for unvisited pages, add hover-state rules).

## Return to Verification

After generation completes, the agent returns to `verification.md` step 5 (determine verification scope) — the same browser session continues; specs and design.md now exist on disk.

**IMPORTANT:** If the original user request was unqualified ("verify [url]"), flow generation and verification is ALSO required. Proceed to `references/flow_generation.md` before writing the combined report. Visual generation alone does not satisfy an unqualified "verify" request.
